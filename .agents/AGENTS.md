# قواعد وإرشادات العمل على المشروع (Project-Scoped Rules)

* **الحفاظ على سلامة واستقرار النظام (Code Integrity & Stability)**: 
  عند إضافة أي ميزة جديدة، أو تعديل، أو تحسين، أو إصلاح لأي عطل، يجب الحفاظ بشكل صارم على جميع الوظائف البرمجية السليمة القائمة في المشروع. يُمنع منعاً باتاً حذف أو إهمال أي كود، منطق عمل (Business Logic)، أو إعدادات سابقة تؤثر سلباً على استقرار النظام أو وظيفته.
  
* **التحقق المستمر (Continuous Verification)**:
  قبل إنهاء أي مهمة، يجب تشغيل اختبارات الوحدة (`npx vitest run`) والتحقق من سلامة البناء الكلي للمشروع (`npm run build`) لضمان عدم حدوث أي تراجع (Regressions).

* **الموافقة المسبقة على التعديلات (Pre-Approval for Modifications)**:
  عندما يطلب المستخدم أي تعديل، أو حذف، أو تحسين في المشروع، يجب على المساعد تقديم مقترح واضح ومفصل بالخطوات والتغييرات التي سيقوم بها، والانتظار للحصول على موافقة صريحة من المستخدم (بالقبول أو الرفض) قبل البدء في تنفيذ أي تعديل على الكود.

* **تطابق منطق الواجهات والمحاسبة (UI & Core Logic Alignment)**:
  يجب التأكد دائماً من أن جميع البطاقات الإحصائية والرسوم البيانية والمؤشرات المعروضة في واجهات المستخدم (UI Components) مرتبطة ومحسوبة مباشرة من محركات الحسابات الأساسية في المشروع (مثل محرك الحسابات الانتخابية ومحرك المؤشرات)، وتجنب استخدام المتغيرات التبسيطية أو الحشوات المؤقتة (Placeholders) التي قد تؤدي لتطابق غير منطقي في الأرقام أو تجاوز معاملات الكفاءة.

* **معايير المعمارية وحماية البيانات (Architectural & Data Protection Standards)**:
  * **النسخ الاحتياطي التلقائي والاستعادة الكاملة:** يجب الحفاظ على تزامن ومزامنة كافة جداول قاعدة البيانات بما في ذلك المنافسين (`Competitor`) واتجاهات الرأي (`SentimentTrend`) في النسخ الاحتياطي وإبطال الكاش فور الاستعادة.
  * **معالجة الأخطاء والمدققات:** يجب استخدام عقود التحقق من المدخلات عبر Zod Enums لجميع حقول الحالات والنوعية قبل المعاملات على قاعدة البيانات لمنع تسلل قيم غير مطابقة.
  * **آلية الصمود والجدولة:** يجب تنفيذ آلية المحاولة التكرارية الأسيّة (`retryWithBackoff`) في كافة المهام الخلفية الحيوية.
  * **المرجعية المعمارية:** يجب على المساعد الرجوع التلقائي لدليل أنماط الوكيل المعتمد في المشروع `docs/AGENT_PATTERNS.md` كدستور إرشادي لتصميم المهارات والحلول البرمجية وتوجيه المهام المعقدة.
  * **حماية استمرارية قاعدة البيانات وعزل بيئات التشغيل:** يلتزم الوكيل بالحفاظ الكامل على سلامة واستمرارية البيانات الحية وتجنب استخدام قواعد البيانات المؤقتة (SQLite dev.db) في بيئة الإنتاج لكونها تفقد محتواها عند إعادة التشغيل أو النشر التلقائي على Railway، والتأكد دائماً من أن بيئة الإنتاج مربوطة بقاعدة بيانات PostgreSQL حية ومستمرة. يُمنع منعاً باتاً تشغيل أي أمر تدميري لقاعدة البيانات (مثل prisma db push --force-reset أو prisma db push مع قبول فقدان البيانات) في أي بيئة تطوير أو إنتاج دون وجود نسخة احتياطية حديثة ومؤكدة والحصول على موافقة خطية صريحة ومباشرة من المستخدم.

* **وقاية وتأمين واجهات المستخدم (Defensive Frontend & API Contracts)**:
  * **البرمجة الوقائية:** يجب دائماً التحقق من وجود الحقول والمصفوفات القادمة من الخلفية قبل قراءة خصائصها في الواجهات (مثل استخدام `searchResults.voters?.length ?? 0` أو `(searchResults.voters || []).length`) لتجنب انهيار شجرة مكونات React بالكامل في بيئة التشغيل.
  * **تطابق الأنواع المشتركة:** يجب توحيد وتعريف واجهات TypeScript مشتركة للبيانات المتبادلة بين الـ API والواجهات لمنع حدوث اختلافات في التسميات أو البنية.
  * **اختبارات التكامل للواجهات:** يجب تدعيم التغييرات الجوهرية باختبارات تحاكي حركة المستخدم وتفاعله مع العناصر للتأكد من سلامة ربطها بالخلفية.

* **الرفع التلقائي للتحديثات (Automatic Push to Git)**:
  بعد الانتهاء من تنفيذ أي خطة تعديل أو إصلاح معتمد واجتياز فحوصات البناء والاختبارات بنجاح، يجب على المساعد عمل commit ورفع (push) التعديلات تلقائياً إلى مستودع GitHub لضمان تحديث بيئة الإنتاج باستمرار.
