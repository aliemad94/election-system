# 📚 دليل قاعدة المعرفة لتطوير وإدارة نظام الانتخابات
## Election System Knowledge Base (KB) for AI Agents

هذا المستند هو المرجع المعرفي والمهاري الإلزامي لأي وكيل ذكاء اصطناعي يعمل على تطوير أو صيانة هذا المشروع. يجب الالتزام الصارم بما ورد فيه لضمان سلامة البنية التحتية البرمجية ودقة البيانات.

---

## 1. محاور حماية البيانات وسلامة النظام

### 1.1 النسخ الاحتياطي التلقائي (Automatic Database Backup)
* **المسار الافتراضي:** يتم حفظ ملفات النسخ الاحتياطي في مجلد `backups/` في جذور المشروع بصيغة JSON.
* **الأتمتة والجدولة:**
  * يتم فحص وجود نسخة احتياطية لليوم الحالي عند بدء تشغيل السيرفر وكل 4 ساعات عبر الجدولة الخلفية (`scheduler.ts`).
  * يتم تنظيف النسخ الاحتياطية تلقائياً والإبقاء على آخر **7 نسخ فقط**.
* **البيانات المشمولة:** يجب أن يشمل النسخ الاحتياطي كامل جداول قاعدة البيانات بما في ذلك المنافسين (`Competitor`)، واتجاهات الرأي (`SentimentTrend`)، وسجلات الثقة والتنبيهات وحملات الرسائل النصية، والتحكم بالوصول.

### 1.2 الاستعادة الآمنة والذرية (Atomic Transactional Restore)
* **المعاملات الذرية:** تتم الاستعادة بالكامل بداخل معاملة واحدة (`prisma.$transaction`) لضمان عدم وجود بيانات متناثرة في حال الفشل.
* **ترتيب الحذف والإدراج:** يجب تفريغ الجداول التابعة والمفاتيح الخارجية أولاً، ثم إعادة إدراج البيانات بالترتيب الهرمي الصحيح (المتطوعين -> القبائل -> المفاتيح الانتخابية -> الناخبين -> الخدمات -> المهام -> التنبيهات -> النتائج والمنافسين).
* **إبطال الكاش:** يجب استدعاء دوال إبطال الكاش (`invalidateComprehensiveIndicatorsCache()` و `invalidateIndicatorsCache()`) فور نجاح الاستعادة لإعادة تحميل البيانات الحية إلى الواجهات.

## 2. محركات الحسابات والمؤشرات الانتخابية

يحظر تماماً الحشو ببيانات ثابتة أو افتراضية متفائلة في محرك المؤشرات. يجب ربط وحساب كافة القيم الحيوية من البيانات الفعلية:

### 2.1 مؤشر التغلغل الجغرافي (API - Area Penetration Index)
يقيس مدى توسع ونفاذ الحملة في منطقة معينة:
$$\text{API} = (\text{Neutral Percent} \times 0.3) + (\text{Expansion Percent} \times 0.25) + (\text{KRI} \times 0.25) + (\text{Weighted Score} \times 0.2)$$

### 2.2 مؤشر خسارة الأصوات والإنذار المبكر (EWLI - Early Warning Loss Index)
يقيس مخاطر تسرب الأصوات والتهديدات الميدانية:
$$\text{EWLI} = (\text{Weak Votes Ratio} \times 0.3) + (\text{DRS} \times 0.25) + (\text{Threats} \times 0.2) + (\text{Support Decline} \times 0.15) + (\text{Competitor Strength} \times 0.1)$$
* **قوة المنافسين (`competitorStrength`):** تُحسب استناداً لدرجة `strengthLevel` (من 1 إلى 5) للمنافسين الفعليين المتواجدين في تلك المنطقة مضروبة في 20 لتحويلها إلى نسبة مئوية.

### 2.3 مؤشر القوة الجغرافية (GSI - Geographic Strength Index)
يقيس مدى التغطية والتوازن الميداني:
$$\text{GSI} = (\text{Coverage} \times 0.25) + (\text{Vote Distribution} \times 0.25) + (\text{Weighted Score} \times 0.25) + (\text{Balance} \times 0.25)$$
* **توزيع الأصوات (`voteDist`):** يُحسب بنسبة الأصوات المؤيدة الفعلية من إجمالي أصوات المنطقة.
* **التوازن (`balance`):** يُحسب استناداً للانحراف المعياري لعدد أصوات الناخبين عبر المفاتيح الانتخابية المختلفة لضمان عدم تركز القوة لدى مفتاح واحد.

### 2.4 مؤشر جاهزية يوم الانتخابات (EDRI - Election Day Readiness Index)
$$\text{EDRI} = (\text{Trained Keys \%} \times 0.2) + (\text{High Protection Keys \%} \times 0.2) + (\text{GPS Verified Voters \%} \times 0.2) + (\text{Registry Verified Voters \%} \times 0.2) + (\text{Loyalty} \times 0.2)$$

## 3. حارس التأمين وقواعد المصادقة والـ API

### 3.1 معايير التحقق من المدخلات (Zod Schema Validation)
* يجب تمرير كافة الطلبات الواردة إلى معالجات الـ API الحساسة عبر مخططات التحقق (Schemas) المحددة في `validators.ts`.
* يجب تقييد المدخلات المعروفة باستخدام `z.enum([...])` بدلاً من strings مفتوحة (مثل: `gender`, `status`, `priority`).

### 3.2 معيار حماية العمليات التدميرية (Double Confirmation Action)
* يجب حماية عمليات الحذف الشامل أو إعادة تعيين النظام بالكامل بطلب تأكيد مزدوج نصي يرسله العميل متطابقاً مع كلمة مرور الإجراء (مثل: `CONFIRM_RESET_ALL_DATA`).

## 4. قائمة الفحص الإلزامية قبل تسليم أي تعديل (Checklist)

عند قيامك بأي تعديل، تأكد من تطبيق الآتي بشكل صارم:
- [ ] هل قمت بتشغيل اختبارات الوحدة بالكامل عبر `npx vitest run` وتأكدت من نجاحها؟
- [ ] هل قمت بالتحقق من بناء النسخة النهائية بنجاح عبر `npm run build`؟
- [ ] هل حافظت على سلامة وتماسك واستقرار قاعدة البيانات والملفات ولا توجد تراجعات؟
- [ ] هل تأكدت من عدم استخدام أي placeholders أو أرقام افتراضية تبسيطية بالواجهات؟
- [ ] هل تم تغليف وتأمين كافة العمليات التدميرية بإجراءات تأكيد مزدوجة؟
- [ ] هل تم إبطال الكاش وتحديث المحرك مباشرة بعد استعادة قاعدة البيانات؟
