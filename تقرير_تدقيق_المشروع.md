## 1. ثغرات حرجة (Critical)

### 1.1 أسرار حقيقية مكشوفة في Git ومرفوعة على GitHub
ملف `PROJECT_HANDOFF.md` (متتبَّع في Git، موجود في آخر commit الحالي) يحتوي على:
- كلمة سر حقيقية لقاعدة بيانات Supabase الإنتاجية (`aliemad94@@`) كاملة ضمن `DATABASE_URL`.
- `JWT_SECRET` الإنتاجي الفعلي بالكامل.
- كلمات سر admin/user الافتراضية (`YOUR_ADMIN_PASSWORD` / `YOUR_USER_PASSWORD`).

هذا الملف مرفوع على مستودع GitHub العام `aliemad94/election-system`. حتى أن commit سابق (`ee862ac`/`defcbd8`) أزال `.env` من التتبع، **السر نفسه بقي مكشوفاً عبر `PROJECT_HANDOFF.md`**، فتنظيف `.env` لم يحل المشكلة الجذرية.

**الأثر:** أي شخص يملك صلاحية قراءة المستودع (أو وصل إليه قبل أن يصبح خاصاً) يمكنه الاتصال مباشرة بقاعدة البيانات الإنتاجية وتزييف توكنات JWT صالحة بصلاحية ADMIN.

**الإجراء الفوري المطلوب:**
1. تدوير (rotate) كلمة سر Supabase فوراً من لوحة تحكم Supabase.
2. توليد `JWT_SECRET` جديد (هذا يُسجِّل خروج كل الجلسات الحالية تلقائياً — وهو مطلوب).
3. تغيير كلمتي admin/user الافتراضيتين.
4. حذف الأسرار من `PROJECT_HANDOFF.md` (استبدالها بـ `<REDACTED>`) ومن كل تاريخ Git باستخدام `git filter-repo` أو BFG، ثم force-push.
5. إن كان المستودع عاماً (public) فعلاً، التفكير الجاد بأن البيانات قد تكون نُسخت بالفعل من طرف آخر — التدوير غير قابل للتفاوض.

### 1.2 عطل برمجي يمنع نجاح البناء (Build-breaking)
ملف `src/app/api/electoral-keys/route.ts` يحتوي حالياً على **5 بايتات NUL (`\x00`)** في آخر السطر 199. هذا يكسر TypeScript فعلياً:
```
src/app/api/electoral-keys/route.ts(199,1): error TS1127: Invalid character.
```
بما أن `next.config.ts` يفعّل `typescript.ignoreBuildErrors: false`، فإن `npm run build` (وبالتالي أي نشر على Railway) **سيفشل بشكل مضمون** بالحالة الحالية للملف.

ملاحظة مهمة: النسخة المُسجَّلة في آخر commit بـ Git **سليمة** (لا تحتوي NUL bytes) — العطل موجود فقط في نسخة الملف الحالية على القرص، أي أنه تغيّر محلي غير محفوظ. هذا يعني التلف حدث في جلسة العمل الحالية (انظر 1.3).

**الإجراء:** فتح الملف وحذف الأسطر الفارغة/البايتات الزائدة بعد السطر 197 وإعادة حفظه، ثم إعادة تشغيل `npx tsc --noEmit` للتأكد.

### 1.3 فهرس Git تالف + احتمال تعديل خارجي متزامن على الملفات
أثناء التدقيق، تحول `git status` من حالة نظيفة إلى:
```
error: index uses h... extension, which we do not understand
fatal: index file corrupt
```
هذا حدث **بين استدعاءين متتاليين** ضمن نفس الجلسة، بالتزامن مع ظهور بايتات NUL في `electoral-keys/route.ts` (البند 1.2). هذا يشير بقوة إلى أن شيئاً آخر (محرر، أداة مزامنة سحابية كـ OneDrive، عملية بناء، أو أداة AI أخرى) كان يكتب على المجلد بالتوازي مع هذا التدقيق.

ملاحظة: المشروع نفسه يحتوي سكريبت جاهز لهذه المشكلة بالضبط — `fix-and-push.ps1` — مما يدل على أن فهرس Git قد تلف سابقاً أيضاً (مشكلة متكررة، ليست عرضية لمرة واحدة).

**الإجراء:**
- تأكد من إغلاق أي عملية أخرى (محرر/IDE/مزامنة سحابية) تكتب على هذا المجلد أثناء العمل عليه.
- لإصلاح الفهرس الحالي بأمان (الملفات نفسها على القرص لن تتأثر):
  ```
  rm .git/index
  git read-tree HEAD
  git status
  ```
  أو تشغيل `fix-and-push.ps1` الموجود مسبقاً في جذر المشروع.

## 2. ثغرات وعيوب أمنية عالية الخطورة (High)

### 2.1 التحقق من المستخدم في قاعدة البيانات غير موجود فعلياً (مخالف للموثّق)
رسالة commit (`ee862ac`) تقول حرفياً: *"auth-guard.ts: add DB user validation on every protected request (catches deleted or role-changed users that a JWT alone cannot detect)"*. لكن بقراءة `src/lib/auth-guard.ts` الحالي فعلياً:
- `withAuth()` و`getUserFromRequest()` يقرآن فقط من الـ headers (`x-user-id`, `x-user-role`, `x-user-name`) التي يحقنها `proxy.ts`.
- **لا يوجد أي استدعاء لقاعدة البيانات** في `auth-guard.ts`.

النتيجة: مستخدم محذوف، أو تم تغيير دوره من ADMIN إلى OBSERVER، يبقى محتفظاً بصلاحياته القديمة كاملة حتى تنتهي صلاحية JWT (7 أيام — انظر 2.2)، لأنه لا يوجد فحص حي على قاعدة البيانات. هذا الإصلاح الأمني الموثّق في تاريخ Git **غير مطبَّق في الكود الحالي**.

### 2.2 مدة الجلسة 7 أيام لا 8 ساعات كما هو موثّق
`src/lib/auth.ts`: `const TOKEN_EXPIRY = "7d";` — بينما commit أمني سابق يذكر تقليصها إلى 8 ساعات. الكود الحالي يستخدم 7 أيام، أي نافذة استغلال أطول لأي توكن مسروق، خصوصاً مع غياب فحص قاعدة البيانات (2.1).

### 2.3 خط الدفاع الوحيد هو `proxy.ts` + ثقة كاملة بالـ headers
النموذج الأمني الحالي: `proxy.ts` يتحقق من JWT ويحقن هويّة المستخدم في headers، وكل route handler "يتحقق" فقط من نفس هذه الـ headers دون أي تحقق مستقل (لا توقيع، لا قاعدة بيانات). توثيق Next.js نفسه لإصدار 16 يحذّر بوضوح أن طبقة `proxy.ts` **ليست حدوداً أمنية (not a security boundary)** — يجب أن يحتوي كل Route Handler على تحقق هوية مستقل خاص به. حالياً هذا غير متوفر؛ كل الحماية تعتمد بالكامل على افتراض أن `proxy.ts` ينفّذ دائماً قبل كل طلب.
هذا النمط مقبول فقط بافتراض عدم وجود طريقة للوصول إلى الخادم متجاوزةً `proxy.ts` — وهو ما يجعل أي ثغرة مستقبلية في طبقة الـ proxy (مثل CVE-2025-29927 الذي سمح بتجاوز middleware في النشر الذاتي self-hosted عبر header مخصص، أو CVE-2026-44574 الأحدث) تعني بايباس كامل للمصادقة لكل الـ API.
**التوصية:** إضافة تحقق توكن مستقل (jose) داخل `withAuth()` نفسه، بدل الاعتماد فقط على الـ headers المحقونة، بالإضافة لفحص قاعدة بيانات (إعادة تطبيق 2.1).

### 2.4 كلمة سر افتراضية صامتة (Silent Fallback) في `seed.js`
```js
const adminPassword = process.env.ADMIN_PASSWORD || 'YOUR_ADMIN_PASSWORD';
```
التعليق أعلى السطر يقول "Read passwords from environment variables ONLY - no hardcoded defaults" — وهذا **غير صحيح فعلياً**؛ يوجد fallback صريح لكلمة سر معروفة الآن للعامة (مذكورة في PROJECT_HANDOFF.md المرفوع على GitHub). إن نُشر النظام مرة بدون ضبط `ADMIN_PASSWORD` في Railway (خطأ تهيئة بسيط جداً)، سيُنشأ حساب ADMIN بكلمة السر العامة المعروفة دون أي تحذير ظاهر.

### 2.5 `mustChangePwd` لا يُفعَّل فعلياً
كل المستخدمين الافتراضيين يُنشأون بـ `mustChangePwd: true`، والقيمة تُرجَع في رد تسجيل الدخول، لكن **لا يوجد أي فرض من جهة الخادم** يمنع استخدام الـ API طبيعياً قبل تغيير كلمة السر. الفحص الوحيد هو علم (flag) يُفترض أن الواجهة الأمامية تتعامل معه — وهو غير كافٍ كحماية حقيقية، إذ يمكن لأي عميل (curl/Postman) تجاهله بالكامل ومتابعة استخدام الحساب بكلمة السر الافتراضية للأبد.

## 3. مشاكل وتحذيرات متوسطة (Medium)

### 3.1 ثغرات في التبعيات (npm audit)
10 ثغرات بدرجة "moderate"، جميعها transitive عبر حزم غير أساسية:
| الحزمة المتأثرة | السبب | الإصلاح |
|---|---|---|
| `js-yaml` (عبر `@mdxeditor/editor`) | DoS عبر merge keys | يتطلب ترقية كبرى لـ `@mdxeditor/editor@4` |
| `postcss` (عبر `next`/`next-auth`/`next-intl`) | XSS في إخراج CSS | يتطلب تنزيل `next` لإصدار قديم — **غير منطقي، تجاهل** |

لا يوجد إصلاح آمن بدون "breaking changes". الأولوية الفعلية الأهم هنا هي البند التالي (3.2) لأنه يحل جزءاً من هذه التبعيات تلقائياً.

### 3.2 حزمة `next-auth` مثبّتة وغير مستخدمة فعلياً
`package.json` يضم `next-auth@^4.24.11`، لكن لا يوجد أي استيراد لـ `next-auth` في `src/` بالكامل — النظام يستخدم تنفيذاً خاصاً بـ `jose` فقط. هذه التبعية (وتبعياتها المتعددة) تضيف وزناً وسطح هجوم بلا فائدة، وهي نفسها مصدر بعض ثغرات `postcss`/`uuid` أعلاه في جدول npm audit. **التوصية:** حذفها من `package.json` بالكامل.

### 3.3 ملفات schema منفصلة لـ SQLite/PostgreSQL يجب مزامنتها يدوياً
`prisma/schema.prisma` (تطوير، SQLite) و `schema.postgres.prisma` (إنتاج) ملفان مستقلان يُنسخ أحدهما فوق الآخر أثناء بناء Docker. تم التأكد أنهما متطابقان حالياً في الحقول، لكن أي تعديل مستقبلي على حقل في ملف واحد دون الآخر يُنتج عطلاً صامتاً في الإنتاج لا يظهر إلا بعد النشر. هذا خطر بنيوي بسبب غياب مصدر وحيد للحقيقة (single source of truth) لمخطط البيانات.

### 3.4 ملفات ضخمة وغريبة داخل جذر المشروع

## 4. ملاحظات بسيطة / تحسينات (Low)

- لا توجد ثغرات SQL Injection: لم يُعثر على أي استخدام خطر لـ `$queryRaw`/`$executeRaw` (الاستخدام الوحيد هو `SELECT 1` ثابت في فحص الصحة `/api/health`).
- لا يوجد `eval()` في الكود.
- `dangerouslySetInnerHTML` يظهر مرة واحدة فقط في مكوّن `chart.tsx` (مكتبة shadcn القياسية لحقن متغيرات CSS) — لا يعالج مدخلات مستخدم، غير خطر.
- `npx eslint .` لم يُظهر أي تحذيرات — جيد.
- إصدار `next` المثبَّت فعلياً هو `16.2.9`، وهو أحدث من `16.2.6` (الإصدار الذي رقّع 13 ثغرة CVE شملت 3 ثغرات تجاوز مصادقة أُعلن عنها في مايو 2026) — هذا الجزء سليم، فقط تأكد من تثبيت هذا الرقم بدقة (`"next": "16.2.9"` بدون `^`) لمنع تنزيل تلقائي غير مقصود لإصدار أقدم غير مرقّع.
