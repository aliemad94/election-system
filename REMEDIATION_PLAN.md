**بوابة التحقق**:

## المرحلة 1 — تدوير الأسرار (فوري، مستقل عن تنظيف التاريخ) 🔑

**المشكلة P1 (الجزء العاجل)**: القيم المسرّبة قابلة للاستخدام الآن. تدويرها يبطلها فورًا بغض النظر عن التاريخ.

**الإجراء** (**المالك حصريًا** — الوكيل ممنوع من إدخال/رؤية بيانات الاعتماد):
1. توليد `JWT_SECRET` جديد قوي (32+ حرف):
   ```bash
   node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"
   ```
   (الوكيل يوفّر الأمر؛ الناتج يبقى عند المالك فقط.)
2. تغيير كلمة مرور قاعدة Supabase (Dashboard → Database → Reset database password) → ينتج `DATABASE_URL`/`DIRECT_DATABASE_URL` جديدان.
3. تحديث متغيرات Railway: `JWT_SECRET`, `DATABASE_URL`, `DIRECT_DATABASE_URL` بالقيم الجديدة. التأكد من:
   - `BYPASS_AUTH=false`
   - `NODE_ENV=production`
4. تدوير أي سر آخر ظهر تاريخيًا إن وُجد (`BACKUP_ENCRYPTION_KEY`, `TWILIO_AUTH_TOKEN`).
5. إعادة تشغيل خدمة Railway لالتقاط القيم الجديدة.

**بوابة التحقق**:
- تسجيل دخول ADMIN حي ينجح بالأسرار الجديدة → `200` + كوكي.
- أي جلسة/توكن قديم يُرفض (النظام يبطل الجلسات القديمة تلقائيًا عبر `tokenIssuedBefore` عند تغيير كلمة المرور، لكن تدوير `JWT_SECRET` يبطل كل التوكينات فورًا).
- `curl -s -o /dev/null -w "%{http_code}" https://[DOMAIN]/api/health` → `200`.

> ✅ بعد هذه المرحلة يزول **الخطر العملي الفوري** حتى قبل تنظيف التاريخ.

## المرحلة 2 — تنظيف تاريخ Git (مدمّر، بتنسيق) 🧹

**المشكلة P1 (الجزء الجذري)**: القيم لا تزال قابلة للاستخراج من commits قديمة عبر `git log`.

**الإجراء** (الوكيل يجهّز ويرشد؛ **كل أمر مدمّر يحتاج «ابدأ» صريحًا من المالك**):
1. تثبيت الأداة:
   ```bash
   pip install git-filter-repo
   ```
2. تجهيز ملف الاستبدالات `replacements.txt` (القيم الحقيقية المسرّبة → `***REMOVED***`). القيم تأتي من المالك، لا تُكتب في أي ملف مُتتبَّع.
3. تشغيل التنظيف على النسخة المرآتية (لا على العمل مباشرة أولًا للتجربة):
   ```bash
   git filter-repo --replace-text replacements.txt --force
   ```
   بديل لملفات كاملة يجب محوها من التاريخ:
   ```bash
   git filter-repo --path RAILWAY_DEPLOY.md --path PROJECT_HANDOFF.md --path project_full_source.md --invert-paths --force
   ```
4. `force-push` منسّق لكل الفروع وtags (**بأمر المالك**):
   ```bash
   git push origin --force --all
   git push origin --force --tags
   ```
5. إبلاغ كل من يملك clone/fork بإعادة clone (التاريخ القديم لديهم لا يزال يحوي السر).

**بوابة التحقق**:
- clone نظيف جديد + بحث في كامل التاريخ:
  ```bash
  git clone <REPO_URL> verify-clean && cd verify-clean
  git log --all -p | grep -E "LEAK_CHECK" || echo "نظيف ✓"
  ```
- تشغيل `node scripts/verify-security.js` → **0 نتائج تاريخية** (كانت 7).

## المرحلة 4 — سكربت التحقق الحي الآلي 🤖

**المشكلة P2 + P5**: لا آلية سريعة لتأكيد سلامة الإنتاج بعد النشر.

**الإجراء** (الوكيل يكتب `scripts/verify-live.sh`): سكربت يشغّل دفعة واحدة:
1. `GET /` → `200`
2. `GET /api/health` → `200`
3. رؤوس الأمان موجودة (CSP + X-Frame-Options + HSTS)

7. OBSERVER على `/api/voters` → لا يحوي `nationalId` (اختبار عزل الأدوار)

**بوابة التحقق**: السكربت يُنهي بـ `exit 0` وكل فحص PASS. المالك يشغّله بعد النشر ويعطي الوكيل الناتج.
